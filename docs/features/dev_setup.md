## Key Files That Control the Environment

### 1. `.env.local` — the single switch file

This file is **gitignored** and must exist locally. It overrides all environment values.

**Local mode (current default):**

> ⚠️ **Never commit real keys to this doc.** The local Supabase keys are generated by
> `supabase start` — read them with `supabase status`. Keep `RESEND_API_KEY` only in your
> gitignored `.env.local`. The placeholders below are intentional.

```env
VITE_SUPABASE_URL="http://127.0.0.1:54321"
VITE_SUPABASE_PUBLISHABLE_KEY="<local anon/publishable key — run: supabase status>"
VITE_SUPABASE_SERVICE_KEY="<local service_role key — run: supabase status>"

SUPABASE_URL="http://127.0.0.1:54321"
SUPABASE_ANON_KEY="<local anon/publishable key — run: supabase status>"
SUPABASE_SERVICE_ROLE_KEY="<local service_role key — run: supabase status>"

RESEND_API_KEY="<your Resend API key — keep in .env.local only>"

# Ask-a-Question concierge fallback — responder for platform/concierge packages
# that have no creator_id. See "Ask-a-Question concierge routing" below.
# (A plain account UUID, not a credential.)
CONCIERGE_USER_ID="c93d67d1-db67-483c-a1c7-88d75f16131d"

# Analytics / SEO — each activates only when set (src/lib/analytics/init.ts). Set the
# same vars in the Lovable production build env. See Workstream D in docs/todo/todo.md.
# VITE_GA4_MEASUREMENT_ID="G-XXXXXXXXXX"
# VITE_CLARITY_PROJECT_ID="xxxxxxxxxx"
# VITE_GOOGLE_ADS_CONVERSION_LABEL="xxxxxxxx"
# VITE_GSC_VERIFICATION="..."        # optional — GA4-link/DNS is more robust
# VITE_BING_VERIFICATION="..."       # optional — or import the property from GSC
```

**Production mode** (point `.env.local` at prod):
```env
VITE_SUPABASE_URL="https://ktzsgqrqvwtxlimctkaf.supabase.co"
VITE_SUPABASE_PUBLISHABLE_KEY="sb_publishable_i5xwYqNzT3JOevhcl7-J3w_J2oofXm5"
# Do NOT set VITE_SUPABASE_SERVICE_KEY in production frontend builds
# SUPABASE_* vars below are only used by the Node.js server and edge functions
SUPABASE_URL="https://ktzsgqrqvwtxlimctkaf.supabase.co"
SUPABASE_ANON_KEY="<production anon key>"
SUPABASE_SERVICE_ROLE_KEY="<production service role key — keep secret>"
RESEND_API_KEY="<production Resend key>"
# In real production this is a Supabase secret (see below), not a frontend var.
# Only needed here when running `functions serve` against prod data.
CONCIERGE_USER_ID="<prod concierge user uuid — see 'Ask-a-Question concierge routing'>"
```

> To switch to production locally, simply replace the `VITE_SUPABASE_URL` and key values in `.env.local`. Vite hot-reloads on `.env.local` changes — no server restart needed.

### 2. `src/integrations/supabase/client.ts` — frontend Supabase client

```ts
const SUPABASE_URL =
  import.meta.env.VITE_SUPABASE_URL ?? "https://ktzsgqrqvwtxlimctkaf.supabase.co";
const SUPABASE_PUBLISHABLE_KEY =
  import.meta.env.VITE_SUPABASE_PUBLISHABLE_KEY ?? "sb_publishable_i5xwYqNzT3JOevhcl7-J3w_J2oofXm5";
```

- When `.env.local` sets `VITE_SUPABASE_URL`, the frontend hits that URL.
- When `.env.local` does **not** set it (e.g. CI / production build), the hardcoded production values are used.
- **Do not change the hardcoded fallbacks** — they are the production safety net.

### 3. `supabase/config.toml`

```toml
project_id = "ktzsgqrqvwtxlimctkaf"
```

This links the local Supabase Docker stack to the production project for CLI commands (`supabase db push`, `supabase functions deploy`). Do not change it.

## Switching Environments — Step by Step

### → Switch to LOCAL

1. Ensure Docker is running.
2. Start local Supabase stack:
   ```powershell
   supabase start
   ```
   First run downloads images (~2 min). Subsequent runs are instant.
3. Confirm `.env.local` has local values (see above).
4. Start the frontend:
   ```powershell
   npm run dev
   ```
5. Serve edge functions (required for any `supabase.functions.invoke(...)` call):
   ```powershell
   supabase functions serve --env-file .env.local
   ```
   Edge functions are available at `http://localhost:9999/<function-name>`.
6. Verify: open `http://127.0.0.1:54323` — Supabase Studio should load with local data.

### → Switch to PRODUCTION (read-only testing)

1. In `.env.local`, replace the `VITE_SUPABASE_URL` and key values with production values.
2. Vite auto-reloads — no restart needed.
3. The frontend now reads/writes production data.  
   ⚠️ **Do not run `supabase db reset` or migrations while pointed at production.**

## Local Supabase Keys

Keys are generated fresh by `supabase start` each time and printed to the terminal. They can also be retrieved at any time:

```powershell
supabase status
```

The new-format keys look like:
- Publishable (anon): `sb_publishable_<token>`
- Secret (service role): `sb_secret_<token>`

Copy these into `.env.local` whenever you see `401 Unauthorized` or auth errors after a `supabase stop` / `supabase start` cycle.

> **Note:** The warnings `Env name cannot start with SUPABASE_, skipping` when running `supabase functions serve --env-file .env.local` for `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` are harmless — the edge runtime injects those automatically from the running stack.

## Resetting the Local Database

Wipes all data and re-runs every migration from scratch:

```powershell
supabase db reset
```

Use this when:
- A new migration is added (`supabase/migrations/*.sql`)
- You want a clean slate for testing
- Schema errors appear locally that don't match production

After a reset, any data seeded manually is gone. Re-seed via Supabase Studio (`http://127.0.0.1:54323`) or SQL scripts in `SEED_DATA_SCRIPT.sql`.

## Debugging Edge Functions Locally

1. Start the stack and serve functions (see above).
2. Watch logs in the terminal running `supabase functions serve`.
3. All `console.log` / `console.error` from function code appears there.
4. To test a function directly (bypassing the frontend):
   ```powershell
   curl -s -X POST http://localhost:9999/submit-trip-inquiry `
     -H "Content-Type: application/json" `
     -d '{"email":"test@example.com","tripId":"<uuid>","question":"Test?"}'
   ```
5. Email output during local dev goes to **Inbucket** (not real inboxes): `http://127.0.0.1:54324`

### Common local edge function errors

| Error | Cause | Fix |
|---|---|---|
| `RESEND_API_KEY not set` | Env var missing or `functions serve` started before key was added | Restart `supabase functions serve` after editing `.env.local` |
| `module not found: @supabase/storage-js` | Using `https://esm.sh/` imports | Use `npm:@supabase/supabase-js@2.x.x` specifier instead |
| `401 Unauthorized` | Wrong or stale anon/service key | Run `supabase status`, copy fresh keys into `.env.local` |
| FK constraint violation on `trip_id` | Local DB has no trips seed data | Remove FK from migration or seed trips first |

## Ask-a-Question concierge routing (`CONCIERGE_USER_ID` + package creator backfill)

The "Ask a Question" flow on a trip page turns an inquiry into a real conversation
owned by a **responder**. `submit-trip-inquiry` resolves that responder server-side,
in this order:

1. `packaged_trips.creator_id` — the user who created the package (a `profiles.id`).
2. `packaged_trips.agent_id` → `travel_agents.user_id` — for agent-owned packages.
3. client-supplied `partnerId`, then **`CONCIERGE_USER_ID`** (env var / Supabase secret).

**Platform / "Goldsainte Concierge" packages have no individual creator**, so for them
the only responder is the `CONCIERGE_USER_ID` fallback. If nothing resolves, the lead is
still captured in `pending_inquiries` but **no conversation is created** — you'll see
`responderId: null` in the `submit-trip-inquiry` logs, and the conversation never appears
in the inbox. (The conversation itself is created in the `dm_conversations` /
`direct_messages` tables via `send-direct-message` — the same model the inbox reads.)

### What the concierge user is
A real, monitored account that receives and replies to inquiries on platform packages.
It must be a **different** account than the traveller. Create it through the normal signup
flow so the team has a login.

### Local configuration (already applied)
The seed data ships platform packages with **no `creator_id`**, so Ask-a-Question can't
route until a responder exists. Local was fixed by:

- Repurposing a test account as the concierge —
  `c93d67d1-db67-483c-a1c7-88d75f16131d` (display name "Goldsainte Concierge").
- Backfilling every creator-less package to it:
  ```sql
  update public.packaged_trips
  set creator_id = 'c93d67d1-db67-483c-a1c7-88d75f16131d'
  where creator_id is null;
  ```
- (Optional) the `CONCIERGE_USER_ID` line in `.env.local` above as a belt-and-braces fallback.

> ⚠️ `supabase db reset` wipes both the backfill and the concierge display name. Re-run the
> `update` above (and re-set the display name) after any reset.

### Production setup (when moving to live)
1. **Ensure a concierge account exists** in prod (sign up e.g. `concierge@goldsainte.ai`).
2. **Get its user id** from the live DB — Supabase Dashboard → SQL Editor (project
   `ktzsgqrqvwtxlimctkaf`):
   ```sql
   select p.id, u.email, p.account_type, p.display_name
   from public.profiles p join auth.users u on u.id = p.id
   where u.email = 'concierge@goldsainte.ai';
   ```
3. **Set the secret** — the durable fix; covers every platform package, current and future,
   with no per-package backfill:
   ```powershell
   supabase secrets set CONCIERGE_USER_ID=<uuid> --project-ref ktzsgqrqvwtxlimctkaf
   ```
4. **(Optional) Backfill** existing platform packages for explicit data — preview, then update:
   ```sql
   select id, slug, title, creator_id, creator_type
   from public.packaged_trips where creator_id is null;     -- preview first

   update public.packaged_trips
   set creator_id = '<uuid>' where creator_id is null;
   ```
5. **Deploy** the functions that resolve the responder and create the conversation:
   ```powershell
   supabase functions deploy submit-trip-inquiry --project-ref ktzsgqrqvwtxlimctkaf
   supabase functions deploy send-direct-message  --project-ref ktzsgqrqvwtxlimctkaf
   ```

> The concierge's `account_type` **cannot** be changed via raw SQL — an anti-escalation
> trigger blocks it. If the concierge needs agent/creator capabilities (e.g. the
> "Send a Proposal" button in the inbox), set the role via the `admin-set-user-role` function.

## Deploying to Production

```powershell
# Deploy a single edge function
supabase functions deploy submit-trip-inquiry --project-ref ktzsgqrqvwtxlimctkaf

# Push pending migrations to production DB
supabase db push --project-ref ktzsgqrqvwtxlimctkaf
```

Set production secrets (edge function env vars) via:
```powershell
supabase secrets set RESEND_API_KEY=re_xxx --project-ref ktzsgqrqvwtxlimctkaf
supabase secrets set CONCIERGE_USER_ID=<uuid> --project-ref ktzsgqrqvwtxlimctkaf
```
> See "Ask-a-Question concierge routing" above for how to obtain the concierge UUID.
